# Lessons Learned

ユーザーからの指摘・修正を受けて記録するパターン集。
同じミスを繰り返さないためのルールブック。セッション開始時にレビューすること。

---

## 防止ルール

### ベストプラクティスファースト原則

> **デザイン値（余白・サイズ・ブレイクポイント等）を提案するとき、独自の「最適解」を計算する前に、業界標準のパターンを確認して提示する。**

1. **まず業界標準を調べる** — Material Design / Bootstrap / Tailwind UI 等で定番のパターンを確認
2. **標準パターンを提示する** — 「業界標準はこう。これでいくか、独自に変えるか？」
3. **独自値を出すなら根拠を明示** — 「標準は X だが、Y の理由で Z にしたい」
4. **問題が出たら推測で返さない** — DevTools・ログ・実際の出力で事実を確認してから回答

### Sycophancy 防止ルール

1. **「〜すべき？」にはデータライフサイクルを端から端までシミュレーションしてから答える**
2. **直前5ターンの自分の主張と矛盾していないか確認してから発言する**
3. **ユーザーが何かを提案したとき、「いいね」「その通り」から入らない。** まず「それは本当にこのプロジェクトに必要か？」を5秒考える。不要なら不要と言う。ADR、分割コミット、`:visited` → 全部ユーザーの提案にそのまま乗って、後から「実はオーバーだと思う」と言い出すパターン。最初から言え。

## 失敗事例

### 例1: 実際の値を確認してから仮説を立てる

Realtime 不動 → トークン未設定なのに RLS を疑った。

### 例2: 参照元を実測してから実装

UI リサーチ概念をそのまま実装 → 参照元の実寸を確認せずフル幅バナー。

### 例3: 既存の仕組みを先に確認

既読管理等を自前実装 → ブラウザが `:visited`・bfcache で解決済み。

### 例4: 根本原因はコンテナ幅。引き算で解決

OGP 画像が大きい → 画像に制約を足し続けた。

### 例5: 「誰が・いつ・なぜ必要か」を先に問う

定期実行をサーバー側 cron で設計 → シングルユーザーには不要。

### 例6: Sycophancy（上記防止ルール参照）

IDB 肥大化の設計議論で毎ターン矛盾した回答。

### 例7: 外部リクエストは「最悪の入力サイズ」で考える

OGP 334件同時フェッチでハング。

### 例8: 既存セレクタとの相互作用を全確認

CSS クラス追加で transition 競合 → 連鎖的場当たり修正。

### 例9: DevTools で確認してから修正

Tooltip (0,0) → 仮説未検証で3回修正。CSS animation は transform を上書きする。

### 例10: ベストプラクティスファースト（上記防止ルール参照）

カード余白調整で独自の 20px を提案 → 問題発生 → 場当たり修正 → ユーザーに「レスポンシブで分けるのが普通では？」と指摘されてやっと正解（16→24px）。

失敗の構造:
1. 「8pt グリッドの 2.5 倍 = 20px」という独自計算で提案 → 実際の業界標準は「モバイル 16px / デスクトップ 24px のレスポンシブ切替」
2. 問題が報告されたとき（padding が消えた）推測で「キャッシュでは？」と返した → DevTools 等で実際の状態を確認すべき
3. ユーザーが正解を教えてくれるまで気づかない → 最初から標準を調べていれば一発だった

### 例11: 議論の成果物はファイルに永続化する

セッション内でリリースタスクリストを議論・確定したが、`tasks/todo.md` にもメモリにも書き出さなかった。次のセッションで完全に消失し、ユーザーに「忘れたのか」と指摘される。

**ルール: 議論でリストや決定事項が確定したら、その場で `tasks/todo.md` またはメモリファイルに書き出す。セッションコンテキストは揮発する。**

### 例12: CLAUDE.local.md の記事ネタ自動キャプチャを実行しなかった

`CLAUDE.local.md` に明記されている「完了報告前・設計判断決着時・ハマり解決時に記事ネタチェック → `docs/articles/` に出力」のルールを、複数セッションにわたって無視し続けた。

**ルール: 完了報告する前に必ず `CLAUDE.local.md` の記事ネタ自動キャプチャ基準を確認する。該当すれば聞かずに出力する。これはオプションではなく必須の手順。**

### 例13: 構造問題に対してパディングで対処した

モバイルでフローティング要素（アバター、設定ギア）がカードに被る → 反射的に `pt-14` を足した。実際には `grid-cols-1` にしたことでモバイルレイアウト全体が未設計のまま露出した構造問題であり、パディングでは解決しない。

**ルール: レイアウトの被りが報告されたとき、余白の追加は対症療法。まず「なぜ被るのか」の構造原因（Grid 設計、フローティング要素の配置戦略、レスポンシブ設計の欠如）を特定してから、構造的に正しい対処を計画する。**

### 例14: CSS レイアウト問題を場当たり的に試行して3回失敗した

flex トラック内で裏面の高さがカードを押し上げる問題に対し:
1. `absolute` 内部配置 → Floating UI autoUpdate と干渉して無限ループ
2. `absolute` 兄弟配置 → `<a>` タグに変えたら無限ループ再発
3. `self-start` + `max-height: 100%` → flex の cross-axis では効かない

最終的な正解は `max-h-0 overflow-hidden`（非表示時に潰す）という単純なトグル。最初から「なぜ高くなるのか」（flex の cross-axis が最大の子で決まる）を理解して「非表示時に高さ 0 にすればいい」と考えれば1回で済んだ。

**ルール: CSS レイアウト問題は「何が高さ/幅を決定しているか」を仕様レベルで理解してから手を動かす。推測で position や overflow を試すな。flex の cross-axis、grid の track sizing、absolute の containing block — これらの仕組みを確認してから解法を設計する。**
